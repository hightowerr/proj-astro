# Mental Model: Solution-First Dynamic

## Description
Solution-First Dynamic occurs when someone presents a demand for a solution without understanding the underlying problem. This happens when people fall in love with an idea or solution before identifying what problem it should solve. Examples include clients requesting specific features ("We need to build an app!") without defining the problem, or people proposing solutions based on what they've seen elsewhere without considering if it addresses the actual issue at hand. This approach often leads to implementing solutions that don't address real needs.

## When to Avoid (or Use with Caution)
Use with caution when:
- The solution is well-established for the specific problem type
- Time constraints require immediate implementation of known solutions
- The solution proposer has deep expertise in the problem domain
- The solution is standard practice for the industry or situation
- Stakeholders have already validated the problem-solution fit
- The solution is clearly addressing an obvious, well-understood need
- There is limited time for problem exploration before implementation

## Keywords for Situations
- When stakeholders propose specific solutions without defining problems
- When encountering "solutions in search of a problem"
- When facing requests for specific tools, features, or approaches
- When people want to implement solutions they've seen elsewhere
- When stakeholders are excited about new technology without problem alignment
- When proposals seem to address surface issues rather than root causes
- When solutions are proposed based on trend-following rather than need
- When facing "innovation" that doesn't connect to actual problems

## Thinking Steps
1. Identify when someone presents a solution without clearly stating the problem
2. Ask "What problem would this solution solve?" or "What problem are we addressing?"
3. Explore whether the solution addresses root causes or just symptoms
4. Question assumptions embedded in the proposed solution
5. Look for alternative solutions that might address the same problem better
6. Validate that the stated problem (if any) is the real issue to address
7. Consider if the solution might be masking a different problem
8. Test the solution-problem fit with other stakeholders and real users

## Coaching Questions
- What problem is this solution meant to address?
- How do we know this is the right problem to solve?
- What if the proposed solution is addressing the wrong problem?
- Are there simpler ways to address the underlying issue?
- What would happen if we defined the problem first and then looked for solutions?
- How might someone who doesn't know this solution describe the problem?
- What evidence do we have that this solution addresses the real problem?
- Are we solving a problem that doesn't actually exist?
