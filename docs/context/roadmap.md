## Shaped — Customer Lifecycle

These items emerged from a structured design session (2026-07-20) that resolved 16 design decisions across the merchant and end-customer lifecycles. All decisions are recorded in `CONTEXT.md` (glossary) and `docs/adr/0001-seed-platform-customer-identity.md`. The synthesis page `wiki/synthesis/showup-cross-vertical-reliability-aggregation.md` documents the strategic rationale for the cross-vertical scoring direction.

- **Polar subscription lifecycle** — Wire the complete merchant billing lifecycle via the Polar Better Auth plugin (`@polar-sh/better-auth`, `@polar-sh/sdk`). Scope: (1) Add `subscriptionStatus` enum (`trialing | active | past_due | cancelled | expired`), `trialEndsAt`, `subscriptionEndsAt`, `polarCustomerId`, and `graceEndsAt` columns to the `shops` table. (2) Configure Polar server plugin with `checkout` (single product "ShowUp Pro", slug `pro`, `authenticatedUsersOnly: true`), `portal` (customer self-service for subscription management), and `webhooks` (handlers for `onSubscriptionActive`, `onSubscriptionCanceled`, `onSubscriptionUpdated`, `onOrderPaid`, `onCustomerStateChanged`). (3) Add `polarClient()` to the Better Auth client config. (4) Webhook handlers: `onSubscriptionActive` → set `subscriptionStatus = 'active'`, update `subscriptionEndsAt`; `onSubscriptionCanceled` → set `subscriptionStatus = 'cancelled'`, keep access until `subscriptionEndsAt` (end-of-period cancel); `onOrderPaid` (renewal failure) → set `subscriptionStatus = 'past_due'`, `graceEndsAt = now + 7 days`. (5) Next.js middleware on `/dashboard/*` routes: if `expired` or `past_due` past grace → redirect to `/billing` paywall; if `trialing` and `now > trialEndsAt` → redirect to `/billing`. (6) Booking page gate: if `subscriptionStatus` not in `(trialing, active)` → show "Online booking is temporarily unavailable. Please try again later." (7) Soft lock scope: only blocks new bookings and config changes; all background jobs (`resolve-outcomes`, `send-reminders`, `recompute-scores`, etc.), Stripe webhooks, and manage links continue for in-flight appointments. (8) Reactivation: `onSubscriptionActive` → instant restore, all data preserved, no re-onboarding. (9) Stripe Connect stays active during soft lock — no new payment intents created (soft lock blocks bookings), existing deposits settle normally, refunds still work. (10) Promo codes handled natively by Polar's discount/coupon system — no custom code needed. Past due escalation: day 0 "update your card" email, day 3 "last chance" email, day 7 transition to `expired` + "shop paused" email (all via Resend). Prerequisite: Polar organization created, product configured in Polar dashboard, webhook endpoint registered. Build first — this is the most critical missing piece; without it, merchants use ShowUp for free indefinitely.

- **Merchant lifecycle messaging** — Two categories of merchant-facing email, both via Resend. (A) **Transactional**: welcome on signup, trial-ending-soon (day 12), trial-expired (day 14), payment-failed (day 0 of past_due), last-chance (day 3), shop-paused (day 7), subscription-cancelled confirmation, subscription-renewed receipt. Triggered by Polar webhooks and the onboarding drip job. (B) **Onboarding drip sequence**: 7-email hybrid trigger — time-driven schedule (day 1, 3, 5, 7, 10, 12, 14) with completion gates (skip if step already done). Day 1: welcome + quick-start guide. Day 3: "Connect Stripe" (skip if `stripeOnboardingStatus = 'complete'`). Day 5: "Create your first service" (skip if `eventTypes.count > 0`). Day 7: "Set your cancellation policy" (skip if `shopPolicies` exists). Day 10: "Share your booking link" (skip if any appointments exist). Day 12: "Trial ends in 2 days — subscribe now." Day 14: "Trial expired — subscribe to reactivate." Execution: piggybacked on the existing `resolve-outcomes` cron job (midnight daily) — no new Vercel cron slot consumed. After resolving appointment outcomes, the job queries shops where `subscriptionStatus = 'trialing'`, calculates trial day from `createdAt`, checks setup completion flags, and sends the appropriate email via Resend. Logged to existing `messageLog` table. Prerequisite: Polar subscription lifecycle (above) must ship first — the drip depends on `subscriptionStatus` and `trialEndsAt` fields. Build second — highest-leverage messaging for trial-to-paid conversion.

- **Platform customer identity seed** — Implement ADR-0001. Add a `platformCustomers` table keyed on phone number (UUID `id`, `phone` TEXT UNIQUE, `email` TEXT, `createdAt` TIMESTAMP). Add `platformCustomerId` UUID FK to the existing shop-level `customers` table. On every booking, silently create-or-match a platform identity by phone number. No customer-facing exposure — no login, no "my bookings" page, no cross-shop score display. The table exists to accumulate cross-business attendance data from day one so the data network effect compounds from launch. When cross-vertical scoring is later activated, the switch requires: (1) explicit customer opt-in consent to share reliability score across businesses (GDPR: Phase 1 storage is legitimate interest for fraud prevention/service quality; Phase 2 cross-shop exposure requires consent), (2) a score portability layer surfacing aggregated score + tier to new shops without revealing underlying data from other shops, (3) privacy-preserving aggregation — merchants see score and tier, never raw data from other businesses. Phase 1 legal basis: legitimate interest. Privacy policy must mention platform-level identity from day one. No prerequisites — can ship independently at any time. Build alongside or shortly after the Polar lifecycle — the `platformCustomerId` FK is a single migration and the matching logic is a few lines in the booking creation path.

## Parked

- **Multi-currency / international expansion** — Connect account creation hardcodes `country: "GB"` and `default_currency: "gbp"`. Shop policies default to GBP. Currency symbol maps support EUR/USD already, but the Connect account won't accept non-GBP charges without multi-currency config. If expanding beyond UK: make `country` and `default_currency` dynamic from shop config, add currency selection to onboarding, and handle Stripe's per-country capability requirements. Also: `PLATFORM_MINIMUM_DEPOSIT_CENTS = 100` in `tier-pricing.ts` assumes GBP (100 cents = £1). Must become currency-aware — JPY has no subunit (¥100 ≈ 50p, wrong floor), EUR/USD are similar to GBP (€1/$1, acceptable). Build a `MINIMUM_DEPOSIT_BY_CURRENCY` map when this ships. - **Partial refunds with Connect** — System currently does full refunds only (`payment.amountCents`). Stripe supports partial refunds with `reverse_transfer: true` (proportional reversal). If added: need to design fee reversal behaviour — proportional (Stripe default), full fee return on any refund, or no fee return on partials. Also need UI for merchant to specify refund amount and see the fee impact before confirming.
- **Multi-shop / multi-owner support** — Current model is 1:1:1 (one user → one shop → one Stripe Express account). `shops.ownerUserId` has a unique index; `shops.stripeAccountId` has a partial unique index. If multi-shop is ever needed: each shop needs its own Express account (different bank accounts, tax reporting), the owner unique index must be removed, and the Connect flow needs a shop selector. If multi-owner: need to decide which owner controls the Express account (Stripe KYC is identity-bound). If ownership transfer: Express account can't be swapped — would need a new one. Don't solve any of this until the market demands it.

- **Refund rate monitoring and tripwires** — The platform absorbs ~34p per refunded Connect payment with no monitoring. Four progressive tripwires define when to act: *Tripwire 1 — Visibility* (implement when first Connect refund occurs): log processing fee lost per refund via `console.info` in `stripe-refund.ts`. At current scale, manual monthly SQL query (`count(refunded Connect payments) / count(total Connect payments)`) is sufficient. No dashboards, no tables. *Tripwire 2 — Rate Review* (check quarterly at 5+ merchants): platform-wide refund rate. Threshold: 10% (at 10%, ~21% of gross margin consumed by refund fees). Action: investigate which merchants/segments drive the rate. *Tripwire 3 — Intervention Planning* (check quarterly at 10+ merchants): threshold 20% (60% of the way to 32% break-even). Action: begin shaping a cancellation fee feature — options: flat fee (e.g., 50p), percentage of deposit, or merchant-configurable. Don't ship yet — start the design process. Requires: partial refund capability (already parked in roadmap), schema for `cancellationFeeCents`, merchant settings UI, customer-facing copy changes. *Tripwire 4 — Per-Merchant Outlier* (check monthly at 10+ merchants): individual refund rate > 2× platform average for 2+ consecutive months. Action: investigate the merchant's `cancelCutoffMinutes` settings, customer base, and whether policy guidance would reduce the rate. Prerequisites: Tripwire 1 (must have visibility before setting thresholds). Build progressively — each tripwire is only meaningful after the previous one is in place. The tripwires don't dictate outcomes — they force conscious decisions at predefined signals. - **Percentage-based application fee evaluation** — The flat 50p application fee creates an inverted margin structure: higher deposits have thinner margins because Stripe's percentage-based processing fee (1.4% + 20p UK, 2.9% + 20p intl) scales with deposit amount but the app fee doesn't. At £10 deposit: 34p processing = 16p margin. At £20 deposit: 48p processing = 2p margin. At £30 deposit: 62p processing = **-12p loss per booking**. International cards on £10: 49p processing = 1p margin. A percentage-based fee (e.g., 5% of deposit, minimum 50p) would invert this: £10 = 50p fee vs 34p processing = 16p margin (same). £20 = 100p fee vs 48p processing = 52p margin (dramatically better). £30 = 150p fee vs 62p processing = 88p margin. The margin IMPROVES with higher deposits instead of collapsing. Evaluate when: high-deposit verticals (personal trainers at £20+, health clinics at £30+) represent >20% of merchants, OR when any merchant's average deposit exceeds £15. This is a product/pricing decision, not a refund fix — the refund fee issue is a symptom of the flat fee structure. Prerequisite: refund rate monitoring (Tripwire 1-2 above) to quantify actual margin pressure before changing pricing. Cross-dependency: ~~"No minimum deposit floor" fix~~ (RESOLVED 2026-07-07 — £1 minimum floor exists). The floor was designed for a flat fee model; a percentage model changes the economics below the floor — review floor value if pricing model changes. Analysis: `mcp-go/Mental Models/WorkSpace/26-07-01_06-27-45_stripe_processing_fee_not_recoverable_refund/analysis_report.md` (Margin of Safety — inverted margin structure identified as structural vulnerability).

- **Sign-up → Stripe subscription payment journey** — The platform charges $49/month (or ~$39/month annual) but currently has no billing integration — sign-up creates a Better Auth account and immediately grants full access with no payment step, no trial enforcement, and no subscription lifecycle. This item covers the full merchant billing journey across three sub-flows:

  **1. Sign-up + payment (trial → paid conversion):**
  After Better Auth registration, redirect to a Stripe Checkout Session in `subscription` mode with a 14-day `trial_period_days`. No credit card required during trial (use `payment_method_collection: "if_required"` initially, switching to `"always"` once trial abuse is observed). On trial end, Stripe auto-charges the card on file. Store `stripeCustomerId` on the `users` table and `subscriptionId` + `subscriptionStatus` on `shops` (the billable entity). Webhook: `checkout.session.completed` → link Stripe Customer to user, `customer.subscription.created` → store subscription. Gate `/app` routes behind `subscriptionStatus IN ('trialing', 'active', 'past_due')` — `past_due` gets a grace window (see §2 below). Consider: product & price setup in Stripe (single "ShowUp Pro" product, two prices: monthly $49, annual ~$468), pricing page toggle mapped to `price_id` in checkout, Stripe Customer Portal for self-service card updates.

  **2. Failed payment / dunning:**
  When a subscription invoice payment fails, Stripe emits `invoice.payment_failed`. On first failure: `subscriptionStatus` → `past_due`, show a persistent in-app banner ("Payment failed — update your card to keep ShowUp active") with a link to the Stripe Customer Portal or an in-app card update form. After Stripe's Smart Retries exhaust (default: 4 attempts over ~3 weeks), the subscription moves to `canceled` or `unpaid` depending on Stripe's configured `collection_method` behavior. Webhooks: `invoice.payment_failed` (set `past_due`, send Resend email with payment update link), `customer.subscription.updated` (sync status changes — `past_due` → `active` on recovery, or `past_due` → `canceled`/`unpaid` on final failure). On `canceled`/`unpaid`: revoke access (redirect all `/app` routes to a "reactivate" page), pause cron jobs for the shop (reminders, slot recovery, scoring — these consume Twilio/compute resources with no revenue), and send a final "your account has been deactivated" email. Preserve all data — no deletion.

  **3. Cancellation (voluntary churn):**
  Merchant initiates cancellation via Settings → Billing or Stripe Customer Portal. Two options: (a) immediate cancellation — `subscriptionStatus` → `canceled`, access revoked at period end (`cancel_at_period_end: true` so they keep access until paid-through date), or (b) cancel now with immediate effect. Recommended: `cancel_at_period_end: true` (Stripe default, less hostile). Webhooks: `customer.subscription.updated` (when `cancel_at_period_end` flips to true — show "Your plan ends on [date]" banner, offer win-back: "Changed your mind? Reactivate anytime before [date]"), `customer.subscription.deleted` (final cancellation — set `subscriptionStatus` → `canceled`, revoke access, pause shop crons, send exit survey email via Resend). Reactivation: if merchant returns before data retention window (90 days), allow re-subscribe from the reactivate page — create a new Checkout Session linked to the existing `stripeCustomerId` so card and history carry over.

  **Schema changes:** Migration adds `stripeCustomerId: text` to `users`, `subscriptionId: text` + `subscriptionStatus: text` + `subscriptionCurrentPeriodEnd: timestamp` + `cancelAtPeriodEnd: boolean` to `shops`. New webhook route: `api/stripe/billing-webhook/route.ts` (separate from platform-webhook and connect-webhook — different event types, different concerns). New middleware or layout guard: check subscription status before rendering `/app` children.

  **Cross-dependencies:** Stripe Connect onboarding (existing) is orthogonal — a merchant can have an active subscription but incomplete Connect setup. These are independent axes: subscription gates platform access, Connect gates customer payment collection. No circular dependency. Existing billing page (`/app/settings/billing`) currently shows customer-facing payment data (deposits collected, refunds issued) — platform subscription billing should be a separate section or page to avoid confusion between "money your customers pay you" and "money you pay ShowUp."

  **Build when:** Before first paid merchant acquisition push. This is a prerequisite for revenue — without it, the product is permanently free. Priority: HIGH. Estimated scope: ~10 specs across 3 waves (Stripe product/price setup, checkout integration, webhook handlers, access gating, dunning UI, cancellation flow, reactivation).
