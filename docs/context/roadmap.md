## Shaped — Customer Lifecycle

These items emerged from a structured design session (2026-07-20) that resolved 16 design decisions across the merchant and end-customer lifecycles. All decisions are recorded in `CONTEXT.md` (glossary) and `docs/adr/0001-seed-platform-customer-identity.md`. The synthesis page `wiki/synthesis/showup-cross-vertical-reliability-aggregation.md` documents the strategic rationale for the cross-vertical scoring direction.

- **Polar subscription lifecycle** — Wire the complete merchant billing lifecycle via the Polar Better Auth plugin (`@polar-sh/better-auth`, `@polar-sh/sdk`). Scope: (1) Add `subscriptionStatus` enum (`trialing | active | past_due | cancelled | expired`), `trialEndsAt`, `subscriptionEndsAt`, `polarCustomerId`, and `graceEndsAt` columns to the `shops` table. (2) Configure Polar server plugin with `checkout` (single product "ShowUp Pro", slug `pro`, `authenticatedUsersOnly: true`), `portal` (customer self-service for subscription management), and `webhooks` (handlers for `onSubscriptionActive`, `onSubscriptionCanceled`, `onSubscriptionUpdated`, `onOrderPaid`, `onCustomerStateChanged`). (3) Add `polarClient()` to the Better Auth client config. (4) Webhook handlers: `onSubscriptionActive` → set `subscriptionStatus = 'active'`, update `subscriptionEndsAt`; `onSubscriptionCanceled` → set `subscriptionStatus = 'cancelled'`, keep access until `subscriptionEndsAt` (end-of-period cancel); `onOrderPaid` (renewal failure) → set `subscriptionStatus = 'past_due'`, `graceEndsAt = now + 7 days`. (5) Next.js middleware on `/dashboard/*` routes: if `expired` or `past_due` past grace → redirect to `/billing` paywall; if `trialing` and `now > trialEndsAt` → redirect to `/billing`. (6) Booking page gate: if `subscriptionStatus` not in `(trialing, active)` → show "Online booking is temporarily unavailable. Please try again later." (7) Soft lock scope: only blocks new bookings and config changes; all background jobs (`resolve-outcomes`, `send-reminders`, `recompute-scores`, etc.), Stripe webhooks, and manage links continue for in-flight appointments. (8) Reactivation: `onSubscriptionActive` → instant restore, all data preserved, no re-onboarding. (9) Stripe Connect stays active during soft lock — no new payment intents created (soft lock blocks bookings), existing deposits settle normally, refunds still work. (10) Promo codes handled natively by Polar's discount/coupon system — no custom code needed. Past due escalation: day 0 "update your card" email, day 3 "last chance" email, day 7 transition to `expired` + "shop paused" email (all via Resend). Prerequisite: Polar organization created, product configured in Polar dashboard, webhook endpoint registered. Build first — this is the most critical missing piece; without it, merchants use ShowUp for free indefinitely.

- **Merchant lifecycle messaging** — Two categories of merchant-facing email, both via Resend. (A) **Transactional**: welcome on signup, trial-ending-soon (day 12), trial-expired (day 14), payment-failed (day 0 of past_due), last-chance (day 3), shop-paused (day 7), subscription-cancelled confirmation, subscription-renewed receipt. Triggered by Polar webhooks and the onboarding drip job. (B) **Onboarding drip sequence**: 7-email hybrid trigger — time-driven schedule (day 1, 3, 5, 7, 10, 12, 14) with completion gates (skip if step already done). Day 1: welcome + quick-start guide. Day 3: "Connect Stripe" (skip if `stripeOnboardingStatus = 'complete'`). Day 5: "Create your first service" (skip if `eventTypes.count > 0`). Day 7: "Set your cancellation policy" (skip if `shopPolicies` exists). Day 10: "Share your booking link" (skip if any appointments exist). Day 12: "Trial ends in 2 days — subscribe now." Day 14: "Trial expired — subscribe to reactivate." Execution: piggybacked on the existing `resolve-outcomes` cron job (midnight daily) — no new Vercel cron slot consumed. After resolving appointment outcomes, the job queries shops where `subscriptionStatus = 'trialing'`, calculates trial day from `createdAt`, checks setup completion flags, and sends the appropriate email via Resend. Logged to existing `messageLog` table. Prerequisite: Polar subscription lifecycle (above) must ship first — the drip depends on `subscriptionStatus` and `trialEndsAt` fields. Build second — highest-leverage messaging for trial-to-paid conversion.

- **Platform customer identity seed** — Implement ADR-0001. Add a `platformCustomers` table keyed on phone number (UUID `id`, `phone` TEXT UNIQUE, `email` TEXT, `createdAt` TIMESTAMP). Add `platformCustomerId` UUID FK to the existing shop-level `customers` table. On every booking, silently create-or-match a platform identity by phone number. No customer-facing exposure — no login, no "my bookings" page, no cross-shop score display. The table exists to accumulate cross-business attendance data from day one so the data network effect compounds from launch. When cross-vertical scoring is later activated, the switch requires: (1) explicit customer opt-in consent to share reliability score across businesses (GDPR: Phase 1 storage is legitimate interest for fraud prevention/service quality; Phase 2 cross-shop exposure requires consent), (2) a score portability layer surfacing aggregated score + tier to new shops without revealing underlying data from other shops, (3) privacy-preserving aggregation — merchants see score and tier, never raw data from other businesses. Phase 1 legal basis: legitimate interest. Privacy policy must mention platform-level identity from day one. No prerequisites — can ship independently at any time. Build alongside or shortly after the Polar lifecycle — the `platformCustomerId` FK is a single migration and the matching logic is a few lines in the booking creation path.

## Parked

- **Multi-currency / international expansion** — Connect account creation hardcodes `country: "GB"` and `default_currency: "gbp"`. Shop policies default to GBP. Currency symbol maps support EUR/USD already, but the Connect account won't accept non-GBP charges without multi-currency config. If expanding beyond UK: make `country` and `default_currency` dynamic from shop config, add currency selection to onboarding, and handle Stripe's per-country capability requirements. Also: `PLATFORM_MINIMUM_DEPOSIT_CENTS = 100` in `tier-pricing.ts` assumes GBP (100 cents = £1). Must become currency-aware — JPY has no subunit (¥100 ≈ 50p, wrong floor), EUR/USD are similar to GBP (€1/$1, acceptable). Build a `MINIMUM_DEPOSIT_BY_CURRENCY` map when this ships. Prerequisite: "No minimum deposit floor" fix in current-issues (the constant must exist before it can be made currency-aware). Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_17-50-28_no_minimum_deposit_floor/analysis-report.md` (Inversion model — currency mismatch identified as highest-risk failure mode).
- **Partial refunds with Connect** — System currently does full refunds only (`payment.amountCents`). Stripe supports partial refunds with `reverse_transfer: true` (proportional reversal). If added: need to design fee reversal behaviour — proportional (Stripe default), full fee return on any refund, or no fee return on partials. Also need UI for merchant to specify refund amount and see the fee impact before confirming.
- **Multi-shop / multi-owner support** — Current model is 1:1:1 (one user → one shop → one Stripe Express account). `shops.ownerUserId` has a unique index; `shops.stripeAccountId` has a partial unique index. If multi-shop is ever needed: each shop needs its own Express account (different bank accounts, tax reporting), the owner unique index must be removed, and the Connect flow needs a shop selector. If multi-owner: need to decide which owner controls the Express account (Stripe KYC is identity-bound). If ownership transfer: Express account can't be swapped — would need a new one. Don't solve any of this until the market demands it.
- **Stripe MCC feedback loop** — After a merchant completes Express onboarding, the platform has no visibility into whether Stripe accepted or overrode the MCC it assigned. Stripe may auto-assign a different MCC during verification, creating a silent discrepancy between what the platform thinks the merchant's classification is and what Stripe actually uses. When built: on `account.updated` webhook (after `details_submitted` flips true), retrieve the account via Stripe Accounts API, compare `business_profile.mcc` with the platform's `getMccForBusinessType(shop.businessType)` mapping, and log a warning on mismatch. Store Stripe's assigned MCC in `shops` metadata for reference. Prerequisite: "MCC hardcoded to 7241" fix — **SHAPED** (`docs/shaping/MCC-hardcoded/`, 5 specs, 3 phases). Build when merchant count exceeds ~50, or immediately if any MCC-related decline/chargeback signal appears. Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_15-42-56_mcc_hardcoded_multi_vertical_risk/analysis_report.md` (Feedback Loops gap — cross-cutting finding across all three models).
- **Codebase-wide businessType integration audit** — **COMPLETED as spec 05** in `docs/shaping/MCC-hardcoded/specs/05-codebase-businesstype-audit.md`. Audit found MCC hardcoding is the **only functional integration gap**. Email templates, SMS, booking pages, and landing page are all vertical-agnostic (reference `shop.name` and generic service language). No other hardcoded beauty/hair/barbershop assumptions found. The pattern ("scope expansion without integration audit") remains valid to watch when new verticals are added, but current blast radius is limited to MCC only.
- **Merchant notification on Connect verification complete** — When Stripe confirms a merchant's Express account (`account.updated` with `charges_enabled && details_submitted`), the webhook updates the database but sends no notification. The merchant must refresh the settings page to see the Connected state. For the ~20% whose verification exceeds 60 seconds (who see the `"still-verifying"` fallback with "you can close this page"), there is no callback to tell them verification finished. When notification infrastructure matures: send an email or SMS to the merchant on verification complete ("Your payments are connected — deposits are now live on your booking page"). Prerequisite: booking confirmation email capability (currently SMS-only) so the email infrastructure is proven. Low urgency — merchants typically check back within hours, and the dashboard shows the Connected view on next visit. Build when a second notification use case for merchants appears (disputes, payout failures, review requests) to justify the channel. Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_16-31-13_auto_poll_timeout_no_fallback/analysis_report.md` (Trust model — "no push notification to the merchant when verification completes").
- **Reinstatement recovery scan** — When `account.updated` restores `charges_enabled` on a previously-suspended account, scan for payments flagged with `transferHeld` metadata and either attempt a manual transfer via the Stripe Transfers API or surface an admin action. Low urgency at current volume — manual Stripe Dashboard intervention is sufficient. Prerequisites: "In-flight payments during Connect suspension" P2 (creates the `transferHeld` flags this would scan for) and "Platform webhook unaware of Connect transfers" P1 (subscribe to `transfer.failed`/`transfer.created` for belt-and-suspenders coverage). Build when a second merchant experiences the race condition. Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_13-38-53_inflight_payment_connect_suspension/analysis_report.md` (Feedback Loops model — "closing the reconciliation loop").
- **Webhook config validation in CI/deployment** — Add a pre-deployment or CI step that validates Stripe webhook endpoint configuration against handler whitelists. Each webhook route declares the event types it handles (e.g., platform: `payment_intent.succeeded`, `payment_intent.payment_failed`, `payment_intent.canceled`; Connect: `account.updated`, `transfer.created`, `transfer.failed`). The validator queries the Stripe Webhooks API, compares registered event types per endpoint against the declared whitelist, and fails the check if an event type is registered on the wrong endpoint or missing from the correct one. Catches misconfiguration before it causes silent event drops. Prerequisite: "Webhook event type misconfiguration" fix in current-issues must ship first (the `console.warn` provides runtime detection; this adds pre-deployment prevention). Build when webhook complexity increases — currently 2 endpoints with 4 combined event types; justified at 3+ endpoints or 6+ event types, where manual Stripe Dashboard auditing becomes error-prone. Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_17-18-28_webhook_event_misconfiguration_silent_drop/analysis-report.md` (Margin of Safety §3 "Prevention" tier and Resilience Engineering "Anticipating" potential — currently 0/3, this would raise it).
- **Shared webhook unexpected-event utility** — Extract a reusable `warnUnexpectedEvent(event: Stripe.Event, endpoint: string)` helper that both webhook routes call for unhandled event types. Currently two inline `console.warn` calls (one per webhook) — acceptable at two call sites, but a shared utility prevents log format drift and ensures consistent structured metadata (`eventType`, `eventId`, `endpoint`) across all webhook endpoints. Build when a third webhook endpoint is added (e.g., for Stripe Billing, Stripe Radar, or additional Connect event types on a dedicated endpoint). Do NOT extract prematurely — two call sites is below the abstraction threshold. Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_17-18-28_webhook_event_misconfiguration_silent_drop/analysis-report.md` (§6 Cross-Dependencies — "justified at 3+ webhook endpoints").
- **Misrouted recovery deposit audit** — Before the "Slot recovery bypasses Connect guard" fix shipped, `acceptOffer()` hardcoded `paymentsEnabled: true`, creating PaymentIntents on the platform account when Connect wasn't complete. Any slot recovery bookings during a merchant's Connect setup window produced deposits in the platform's Stripe account with no `transfer_data` — money went to the platform, not the merchant, with no automatic forwarding mechanism. After the fix ships, run a one-time audit: query `appointments` where `source = 'slot_recovery'` AND `depositSkipped IS NULL` AND the associated `payments.metadata` lacks `connectedAccountId`. Any hits represent platform-account deposits that should have gone to the merchant — refund manually from the platform Stripe account. If the merchant never completed Connect (abandoned), refund to the customer. Not an ongoing concern — the fix prevents recurrence. Low urgency: unlikely at current scale (single merchant, slot recovery may not have fired during Connect setup window). Prerequisite: "Slot recovery bypasses Connect guard" fix in current-issues (the audit is only meaningful after the fix stops producing new misrouted deposits). Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_22-28-37_slot_recovery_bypasses_connect_guard/analysis-report.md` (Inversion §3 — worst case: merchant abandons platform, customer deposit trapped indefinitely).
- **Dispute auto-evidence submission from booking data** — The platform holds structured evidence that could strengthen merchant dispute responses: appointment date/time, service booked, customer contact details, booking confirmation SMS delivery status, cancellation policy version in effect, and any prior SMS communications. The Stripe Disputes API (`stripe.disputes.update`) accepts `evidence` with fields like `customer_name`, `customer_email_address`, `service_date`, `service_documentation`, and `uncategorized_text`. When built: on `charge.dispute.created`, auto-populate an `evidence` payload from the linked appointment/payment record and submit via the Disputes API. The merchant still owns the dispute (business of record via `on_behalf_of`), but the platform pre-fills evidence the merchant would struggle to gather from the Express Dashboard alone. Requires: resolving `charge.dispute.created` → payment → appointment lookup chain (P2 in current-issues creates this), Stripe Disputes API integration, and a decision on whether to auto-submit or draft-and-notify (auto-submit is simpler but removes merchant control; draft-and-notify requires in-app dispute review UI). Build when dispute volume exceeds ~5/quarter or a merchant reports losing a dispute they could have won with booking evidence. Prerequisite: "Disputes attributed to merchant" P2 (webhook subscription) and P3 (notification email) in current-issues — detection and notification must exist before evidence submission makes sense. Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_22-41-07_stripe_dispute_visibility/analysis_report.md` (Asymmetric Information §4 — "Signaling" strategy includes evidence as a future layer).
- **In-app dispute management flow** — Full dispute lifecycle management within the app: dispute inbox/list, per-dispute detail view with timeline (opened → evidence submitted → won/lost), evidence upload UI, response deadline countdown, and resolution outcome. Currently parked because: (1) the Express Dashboard already provides this functionality, (2) building a parallel dispute management UI is a significant scope increase, and (3) at current volume (~1 merchant), the probability of disputes is low. When built: dispute list page under `/app/disputes` or within the appointments section, dispute detail view showing Stripe dispute data (amount, reason, deadline, status), evidence submission form, and resolution history. Requires: `disputes` table or `disputeId`/`disputeStatus`/`disputeReason`/`disputeDeadline` columns on the payments table, webhook handlers for `charge.dispute.updated` and `charge.dispute.closed`, and Express Dashboard link as fallback for advanced actions. Build when merchant count exceeds ~20 or support tickets about dispute confusion exceed ~3/quarter — the Express Dashboard is sufficient at low scale, but becomes a UX dead-end at higher volumes where merchants expect all business management to happen in one place. Prerequisite: "Disputes attributed to merchant" P2 and P4 in current-issues (webhook subscription and payment card modifier — the minimum dispute state persistence).
- **Dispute rate monitoring and alerting** — Track per-merchant dispute rates and alert when they approach Stripe's thresholds. Stripe monitors dispute rates at the platform level — a platform with >1% dispute rate across all connected accounts risks review or suspension. Individual connected accounts with high dispute rates face individual suspension (`charges_enabled` revoked). When built: calculate rolling 90-day dispute rate per merchant and platform-wide, alert (email to platform admin) when any merchant exceeds 0.5% (early warning before Stripe's ~1% threshold), and surface dispute rate in the merchant's dashboard with contextual guidance ("Your dispute rate is [X]% — below 1% is healthy"). Requires: dispute event persistence (from P2), a scheduled job or query for rate calculation, and admin notification infrastructure. Build when platform transaction volume exceeds ~1000/month/merchant (below this, individual disputes are handleable case-by-case). Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_22-41-07_stripe_dispute_visibility/analysis_report.md` (Skin in the Game §5 — "Design incentives for tail risks" prescribes long-horizon monitoring).
- **Refund rate monitoring and tripwires** — The platform absorbs ~34p per refunded Connect payment with no monitoring. Four progressive tripwires define when to act: *Tripwire 1 — Visibility* (implement when first Connect refund occurs): log processing fee lost per refund via `console.info` in `stripe-refund.ts`. At current scale, manual monthly SQL query (`count(refunded Connect payments) / count(total Connect payments)`) is sufficient. No dashboards, no tables. *Tripwire 2 — Rate Review* (check quarterly at 5+ merchants): platform-wide refund rate. Threshold: 10% (at 10%, ~21% of gross margin consumed by refund fees). Action: investigate which merchants/segments drive the rate. *Tripwire 3 — Intervention Planning* (check quarterly at 10+ merchants): threshold 20% (60% of the way to 32% break-even). Action: begin shaping a cancellation fee feature — options: flat fee (e.g., 50p), percentage of deposit, or merchant-configurable. Don't ship yet — start the design process. Requires: partial refund capability (already parked in roadmap), schema for `cancellationFeeCents`, merchant settings UI, customer-facing copy changes. *Tripwire 4 — Per-Merchant Outlier* (check monthly at 10+ merchants): individual refund rate > 2× platform average for 2+ consecutive months. Action: investigate the merchant's `cancelCutoffMinutes` settings, customer base, and whether policy guidance would reduce the rate. Prerequisites: Tripwire 1 (must have visibility before setting thresholds). Build progressively — each tripwire is only meaningful after the previous one is in place. The tripwires don't dictate outcomes — they force conscious decisions at predefined signals. Analysis: `mcp-go/Mental Models/WorkSpace/26-07-01_06-27-45_stripe_processing_fee_not_recoverable_refund/analysis_report.md` (Tripwire model — four progressive thresholds).
- **Percentage-based application fee evaluation** — The flat 50p application fee creates an inverted margin structure: higher deposits have thinner margins because Stripe's percentage-based processing fee (1.4% + 20p UK, 2.9% + 20p intl) scales with deposit amount but the app fee doesn't. At £10 deposit: 34p processing = 16p margin. At £20 deposit: 48p processing = 2p margin. At £30 deposit: 62p processing = **-12p loss per booking**. International cards on £10: 49p processing = 1p margin. A percentage-based fee (e.g., 5% of deposit, minimum 50p) would invert this: £10 = 50p fee vs 34p processing = 16p margin (same). £20 = 100p fee vs 48p processing = 52p margin (dramatically better). £30 = 150p fee vs 62p processing = 88p margin. The margin IMPROVES with higher deposits instead of collapsing. Evaluate when: high-deposit verticals (personal trainers at £20+, health clinics at £30+) represent >20% of merchants, OR when any merchant's average deposit exceeds £15. This is a product/pricing decision, not a refund fix — the refund fee issue is a symptom of the flat fee structure. Prerequisite: refund rate monitoring (Tripwire 1-2 above) to quantify actual margin pressure before changing pricing. Cross-dependency: "No minimum deposit floor" fix in current-issues (the £1 minimum floor was designed for a flat fee model; a percentage model changes the economics below the floor). Analysis: `mcp-go/Mental Models/WorkSpace/26-07-01_06-27-45_stripe_processing_fee_not_recoverable_refund/analysis_report.md` (Margin of Safety — inverted margin structure identified as structural vulnerability).
- **Configurable service durations** — The `durationMinutes % slotMinutes === 0` constraint in `services/actions.ts` forces service durations to be multiples of the grid cadence — a Procrustean artifact from the beauty-wellness origin. The booking system already handles arbitrary durations (`booking.ts`, `appointments.ts`, `booking-form.tsx` — slot filtering, calendar events, conflict detection all use `effectiveDuration` independent of grid). The real blocker is application-level validation, not the domain model. **Fix (4 changes, no grid cadence migration):** (1) Remove grid-multiple validation in `validateDuration` — replace `durationMinutes % slotMinutes !== 0` with bounds check `durationMinutes >= 5 && durationMinutes <= MAX`, (2) replace duration dropdown with bounded number input (`min=5, max=480, step=5`) in both `service-editor-form.tsx` and `add-service-step.tsx`, (3) raise `MAX_SERVICE_DURATION_MINUTES` from 240 to 480 in `constants.ts`, (4) add DB upper-bound check constraint on `event_types.durationMinutes` (`<= 480`) for defense-in-depth. Does NOT require expanding the grid cadence set `[15, 30, 45, 60, 90, 120]` or any migration to `booking_settings` — grid cadence and service duration are orthogonal (grid controls when appointments start; duration controls when they end). Buffer options `[0, 5, 10]` unchanged — separate concern with different UX risk profile. Business-type-specific duration defaults deferred — ship the decoupling first, set defaults from real merchant data instead of guessing. **UX note:** when a merchant creates a service whose duration greatly exceeds the grid cadence, surface guidance to consider adjusting the grid. Build when: a new vertical (consulting, workshops, retreats) onboards, or when any merchant requests a non-multiple duration. Prerequisites: none. Analysis: `mcp-go/Mental Models/WorkSpace/26-07-17_16-21-20_increase_slot_times_configurable_duration/analysis-report.md` (9 models: Identifying Prior Questions, Procrustean Bed, Path Dependence, False Trade-offs, Via Negativa, Leverage Points, Second-Order Thinking, Margin of Safety, Necessity & Sufficiency).
- **Google Meet consultation option** — Add a Google Meet video call link to bookings so consultants and other remote-service providers can offer digital consultations. For MVP: when a merchant enables "virtual appointment" on a service type, a Google Meet link is auto-generated via the Google Calendar API's `conferenceData` parameter and included in the booking confirmation SMS/email and the customer manage page. The existing Google Calendar integration (`src/lib/google-calendar.ts`) already creates calendar events — adding `conferenceDataVersion: 1` and `conferenceData: { createRequest: { requestId } }` to the event creation call auto-generates a Meet link. Requires: new `isVirtual` boolean on `eventTypes` schema, UI toggle in service settings, Meet link storage on the appointment or calendar event, and SMS/email template changes to include the link. Build when: consultant or coaching verticals represent >10% of merchants, or when a merchant explicitly requests remote appointments. Cross-dependency: Google Calendar OAuth must be connected (existing prerequisite for calendar sync). Prerequisites: none blocking — the Google Calendar integration and SMS/email infrastructure already exist.
- **Homepage update** — The landing page content was last aligned during the Astro→ShowUp rebrand but still reflects launch-era feature copy. Feature sections should be updated to reflect shipped capabilities (dispute visibility, payout status, slot recovery improvements) and remove aspirational claims for unbuilt features (e.g., "win back clients on autopilot" per §4 exclusion #8). Prerequisites: none — all referenced features are already shipped. Build when: preparing for a growth push, or when the landing page becomes the conversion bottleneck (e.g., high traffic but low trial signups).
- **Merchant-configurable deposit minimum** — Add `minimumDepositCents` column to `shopPolicies` so merchants can set their own floor above the platform minimum (`PLATFORM_MINIMUM_DEPOSIT_CENTS = 100`). Subtracted from the "No minimum deposit floor" fix via Via Negativa analysis: no merchant has requested it, the mechanism for higher floors already exists (set higher `depositAmountCents` / `topDepositAmountCents` / `riskDepositAmountCents`), and it creates a precedence question with tier overrides (does minimum override `topDepositAmountCents`? does the tier override win?). Would require: new column + migration, new input field on payment-policy settings UI, validation that `minimumDepositCents ≥ PLATFORM_MINIMUM_DEPOSIT_CENTS`, and a precedence rule for tier interactions. Build only if genuine demand appears — e.g., a merchant wants different deposit amounts per event type but a consistent floor across all of them (the one use case the existing controls don't cover cleanly). Prerequisite: "No minimum deposit floor" fix in current-issues (the platform floor must exist before a merchant-level floor makes sense). Analysis: `mcp-go/Mental Models/WorkSpace/26-06-30_17-50-28_no_minimum_deposit_floor/analysis-report.md` (Via Negativa §6 — "subtract `minimumDepositCents`"; Necessity & Sufficiency §12 — "NOT necessary").
