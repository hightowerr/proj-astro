# Current Issues

### Open

- **Booking nav links are anchors to landing page sections** (`/#how-it-works`, `/#features`, `/#pricing`, `/#faq`). If landing page sections don't have matching `id` attributes, these links will scroll to top of page instead of the intended section. Verify anchor targets exist.
- **CTA hover/active states not implemented** — spec 08 calls for opacity 0.92 on hover and scale(0.985) on active/press. Requires CSS classes or JS state since the button uses inline styles. Consistent with the no-hover pattern across the reskin. Add later if desired.
- **Stripe Elements appearance still uses Deep Ledger** — will be visible after CTA restyle when user reaches payment step. The contrast between the AL-styled form and the dark Stripe embed is now more pronounced. Not covered by any spec. Additionally, `booking-form.tsx` lines 115–150 contain hardcoded dark-theme hex colors in the Stripe appearance JS object that cannot use CSS variables. Needs a dedicated Stripe theme spec.
- **Dashboard `KickerStyle` type still uses `"emerald"` discriminator string** — not a Tailwind class, just a type literal. The actual `bg-emerald-*` classes have been replaced with `var(--al-status-positive-*)`. Consider renaming to `"positive"` for clarity in a future cleanup pass.
- **Dashboard essentials ExternalLink icon shows at fixed 40% opacity** — previously had hover opacity animation (25% → 80%). Removed per Brand Law #12 (deliberately still). May want to revisit the fixed opacity value for visibility.
- **Slot button hover state not implemented** — spec 05 calls for `#eeeeec` bg + lighter border on hover. Skipped to stay consistent with "design change only" and the V1-V3 pattern (no hover states added via JS). Add later if desired.
- **Success state uses Deep Ledger classes and tokens** — `bg-bg-dark-secondary/70`, `text-white`, `text-primary-light` Tailwind classes AND `--color-brand`, `--color-surface-void`, `--color-brand-border` inline tokens in booking-form.tsx lines 880-942. Explicitly excluded from Wave 2 Slice 3 scope. Will need restyling when a confirmation spec is added.
- **`service-card-arrow` hover animation broken** — `service-selector.tsx` line ~126 uses `className="service-card-arrow"` but the parent class changed from `.service-card` to `.al-service-card` in Wave 2 Slice 3. The CSS hover rule (`.service-card:hover .service-card-arrow` at globals.css line 595) no longer matches. Either add `.al-service-card:hover .service-card-arrow { opacity: 1; }` or remove the arrow animation.
- ~~**3 AL tokens incorrectly marked as added**~~ — RESOLVED: `--al-track-eyebrow` (0.2em), `--al-radius-3xl` (24px), `--al-display-lg` (3.5rem) added to globals.css `:root` in Wave 3 Slice 1 (2026-06-21).
- **`bg-al-background` Tailwind class does not work** — no `--color-al-background` mapping exists in `@theme inline`. Spec #12 references this class. Use `bg-al-surface` or `bg-background` instead (both resolve to `#f9f9f7`).
- **BookingNav shows static "Sign in" / "Start free trial"** regardless of session state. The current `SiteHeader` is session-aware (shows "Open App" / "Sign Out" when logged in). Decide if BookingNav needs session awareness.
- **Hardcoded hex values in 5 non-dashboard components** — `attention-required-table.tsx`, `payments-table.tsx`, `curator-chip.tsx`, `customers-editorial.tsx`, `appointments-table.tsx` contain hardcoded hex values (`#ffdbcf`, `#2a170f`, `#fdd8cb`, `#e2bfb3`, `#380d01`, `#572411`) that should be replaced with DS tokens. Out of scope for Spec #11 (dashboard only). See `docs/shaping/home-screen-update/shapes/spike-hex-scope-beyond-dashboard.md`.
- **Deep Ledger `@theme inline` block still in globals.css** — lines 3-160 define the full legacy `--color-*` token set. Not removed in Wave 1 because other components still reference these tokens. Needs a dedicated cleanup wave to audit all remaining `--color-*` consumers and remove the block.
- **Hardcoded rgba hairlines in booking-form.tsx** — 7 instances of `rgba(195,198,209,0.50)` should use `--al-hairline-strong`. Not a `--color-*` violation but a tokenization gap.
- **Hardcoded rgba warning border in conflict-alert-banner.tsx** — `rgba(201,122,42,0.25)` at line 76 should use `--al-status-caution-border` (now defined in globals.css as of Wave 2 Slice 1). Not a `--color-*` violation but a tokenization gap.
- **`--font-display` and `--font-body` legacy aliases still in `@theme inline`** — Wave 1 Slice 1 repointed them to Manrope; Wave 2 Slice 1 removed their usage from `@layer base` and utility classes. They now exist only as `@theme inline` definitions. 13 inline `var(--font-mono)` references across 4 component files (`booking-form`, `manage-booking-view`, `tier-policy-form`, `payment-policy-form`) still read from `@theme`.
