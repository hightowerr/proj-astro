- **Re-engagement email window anchored incorrectly** — The cron query (`connect-reengagement/route.ts:56-59`) anchors the 24-48h window exclusively to `stripeAccountCreatedAt`: `between(shops.stripeAccountCreatedAt, fortyEightHoursAgo, twentyFourHoursAgo)`. The EXISTS gates for services (line 61) and availability (line 62) check presence but can't compensate for an expired window. If the owner creates their Stripe account before setting up services and adds services after 48h, the window has already closed — the email never fires permanently (sent-once design via `connectReengagementSentAt` guard). Root cause (Path Dependence): the anchor was designed for the expected onboarding sequence (services → availability → Stripe), where `stripeAccountCreatedAt` IS the last prerequisite. But the UI guides this order without enforcing it — merchants can click "Payments" in settings nav at any time. The implementation encodes an assumed sequence the system doesn't guarantee. Root cause (Map Is Not the Territory): `stripeAccountCreatedAt` answers "when was Stripe created?" but the email needs "when was the merchant ready for a nudge?" The merchant is ready when ALL prerequisites coexist — the anchor should be the later of Stripe creation and first service creation. Root cause (Identifying Prior Questions): the prior question "when did the merchant become eligible?" decomposes into Q3a (when was Stripe created?) and Q3b (when was first service created?). The eligibility timestamp is `MAX(Q3a, Q3b)`. Currently answered with Q3a alone. The EXISTS gates (Q5) and the timing input (Q3b) share a component (services) but serve different purposes — Q5 validates current state ("do services exist NOW?"), Q3b anchors the window ("when did services first exist?"). Both are needed: a deleted service after the window opens would be caught by Q5 but not Q3b. Spec 16 (`16-email-connect-abandoned.md:23`) contemplated clock resets for re-onboarding but never considered the services-first scenario. **Decision: replace the `between()` anchor with `GREATEST()` — one subquery, one file, no schema changes.** Replace `between(shops.stripeAccountCreatedAt, fortyEightHoursAgo, twentyFourHoursAgo)` with a raw SQL clause: `GREATEST(stripeAccountCreatedAt, COALESCE((SELECT MIN(created_at) FROM event_types WHERE shop_id = shops.id AND is_active = true), stripeAccountCreatedAt)) BETWEEN fortyEightHoursAgo AND twentyFourHoursAgo`. The `COALESCE` fallback to `stripeAccountCreatedAt` handles the case where no active services exist yet (the EXISTS gate rejects anyway, but prevents `GREATEST()` from returning NULL). EXISTS gates (lines 61-62) remain unchanged — they validate current state, independent of the window anchor. Add comment above the clause: "Window anchor: GREATEST(stripeAccountCreatedAt, firstServiceCreatedAt). The merchant may create Stripe before or after services — the window adapts to whichever prerequisite was met last." **Not chosen**: `connectEligibleAt` column on shops (creates three write-paths — must update on service creation, availability creation, and Stripe account creation; `GREATEST()` computes at read time with no write-path coupling). **Not chosen**: include `shopHours.createdAt` in GREATEST (availability is practically simultaneous with services — same settings visit; adds subquery complexity for near-zero probability). **Not chosen**: extend window beyond 48h (doesn't fix root cause; adaptive anchor handles any delay). **Not chosen**: remove the time window entirely (24-48h delay is intentional behavioral design — sends at the right nudge point). **Architectural principle**: when a system guides a sequence but doesn't enforce it, derived logic must be sequence-agnostic. - **Stripe processing fee not recoverable on refund** — Stripe does not return processing fees on refunds (standard policy since October 2023). Platform absorbs ~34p (1.4% + 20p for UK cards) per refunded Connect payment. With a 50p flat application fee and ~34p processing cost, the platform nets ~16p per booking. On refund (`stripe-refund.ts:199-201`): `reverse_transfer: true` debits the merchant's connected account (merchant made whole), `refund_application_fee: true` returns the 50p to the customer (customer made whole), Stripe keeps the 34p processing fee (platform absorbs). The spec (13-refund-reverse-transfer.md:50) documents this as deliberate: "neither party should be penalised. The platform absorbs the fee loss." Root cause (Moral Hazard): complete risk separation — customers and merchants bear zero cost for cancellations, the platform absorbs all processing fee losses. At low volume this is a feature (generous refunds reduce £15+ chargebacks, improve trust, simplify operations). At scale it becomes structurally unsustainable — **break-even refund rate is ~32%** at standard conditions (16p margin / 50p total = refunds must stay below 32% to remain profitable). Root cause (Margin of Safety): the 50p flat app fee creates an **inverted margin structure** — higher deposits have THINNER margins because Stripe's percentage-based processing scales with deposit amount but the flat fee doesn't. A £10 deposit = 34p processing = 16p margin (32% break-even). A £20 deposit = 48p processing = 2p margin (**6% break-even**). International cards (2.9% + 20p) on £10 = 49p processing = 1p margin (**3% break-even**). The refund cost is a symptom; the flat fee model's mismatch with percentage-based processing is the structural vulnerability. **Decision: accept at current scale, add monitoring via progressive tripwires (parked in roadmap.md).** No fee structure changes, no cancellation fees, no merchant pass-through. The current policy is correct for growth phase — premature fee friction harms adoption more than 34p per refund at low volume. Four tripwires defined (see roadmap.md): (1) visibility — measure the fee loss when first Connect refund occurs, (2) 10% refund rate — investigate what's driving it (quarterly at 5+ merchants), (3) 20% refund rate — start shaping cancellation fee infrastructure (quarterly at 10+ merchants), (4) per-merchant outlier — individual rate > 2× platform average for 2+ months (monthly at 10+ merchants). **Not chosen**: customer cancellation fee (premature — introduces friction in growth phase; build when Tripwire 3 fires). **Not chosen**: merchant pass-through fee (unfair — merchants don't control customer cancellation behavior). **Not chosen**: percentage-based app fee (correct long-term but significant pricing change; flag as pricing review when high-deposit verticals onboard, not as a refund fix). **Not chosen**: reduce refund window (`cancelCutoffMinutes` is merchant-configurable; platform shouldn't override). No existing monitoring: no refund rate metric, no fee loss tracking, no per-merchant refund volume. The dashboard shows "Retained" and "Refunded" totals (`getMonthlyFinancialStats()`) but not the platform's processing fee loss. Not surfaced to merchants (deliberate — platform cost of business). **Cross-dependency**: "Partial refunds with Connect" in roadmap.md would change the fee reversal behavior — must decide: proportional fee return, full return on any partial, or no return on partials. **Deps: independent. Accepted at current scale; no implementation until tripwires fire. Blocks: nothing.** **Analysis**: Mental models report (Moral Hazard, Tripwire, Margin of Safety — all converge) at `mcp-go/Mental Models/WorkSpace/26-07-01_06-27-45_stripe_processing_fee_not_recoverable_refund/analysis_report.md`.
