# Current Issues

### Open

- **Refund state missing from payment card** ŌĆö The 5-state payment card (connect/waived/legacy/skipped/policy) has no "refunded" state. When a Connect payment is refunded (`reverse_transfer: true`, `refund_application_fee: true`), the appointment detail should show the reversal: "Deposit: ┬Ż10, Platform fee: returned, Your payout: ┬Ż0." Currently refund display is handled by the existing `financialOutcome: "refunded"` field but the fee breakdown doesn't reflect the reversal. Need a 6th state or a modifier on the connect/waived states.
- **Platform webhook unaware of Connect transfers** ŌĆö `/api/stripe/webhook` handles `payment_intent.succeeded` but doesn't check transfer status to the connected account. If the transfer fails (account suspended, capabilities revoked), the charge succeeds on the platform but the merchant never gets paid ŌĆö and nobody is notified. Consider subscribing to `transfer.created` and `transfer.failed` events on the Connect webhook for logging/alerting. Also: the webhook doesn't store the fee amount, so booking confirmation SMS/email can't show "Your payout: ┬ŻX.XX" without a Stripe API call.
- **In-flight payments during Connect suspension** ŌĆö If a merchant's `charges_enabled` is revoked while a customer has a live `clientSecret` (mid-checkout), the platform charge may succeed but the automatic transfer to the connected account could fail. The `payment_intent.succeeded` webhook should check the connected account's `charges_enabled` status; if false, hold the funds on the platform account and surface a dashboard action item ("Payment received but transfer paused ŌĆö Stripe is reviewing your account"). Manual transfer after reinstatement. Low probability but real money at stake.
- **Re-engagement email assumes owner reached Stripe** ŌĆö If the redirect to Stripe fails after the Express account is created (browser block, network drop, tab closed), `stripeAccountCreatedAt` is set and the re-engagement clock starts. The email says "You started connecting your Stripe account" but the owner may never have seen the Stripe form. Self-healing (settings page shows "Continue setup" on next visit) but email copy could be softened to "You began setting up deposits" to avoid referencing a Stripe interaction that may not have happened.
- **MCC hardcoded to 7241 (Barber and Beauty Shops)** ŌĆö Express account creation in `create-account/route.ts` hardcodes `business_profile.mcc: "7241"`. Wrong MCC for non-beauty verticals (tattoo parlours, personal trainers, etc.) can cause higher card decline rates and incorrect statement descriptors. Fix when expanding beyond beauty: make MCC configurable per shop ŌĆö either derive from `shops.businessType` or add an `mcc` field to shop settings. Stripe may also auto-assign MCCs during Express verification, partially mitigating this.
- **Free bookings get no confirmation SMS** ŌĆö `sendBookingConfirmationSMS()` is only called from the Stripe `payment_intent.succeeded` webhook. Free bookings (via `/api/appointments`) never trigger it. This means: deposit-waived top-tier customers, all bookings while Connect isn't set up, and policy-no-deposit shops get no SMS confirmation. Pre-existing bug but Connect makes it worse ŌĆö every booking during the merchant's setup window is silent. Fix: call `sendBookingConfirmationSMS()` (or a variant that handles no-payment gracefully) from the `/api/appointments` route after successful appointment creation. The SMS function currently logs `errorCode: "payment_missing"` and bails when there's no payment ŌĆö needs a branch for free bookings that sends a confirmation without the `{{amount}}` variable.
- **Stripe receipt emails may duplicate app confirmations** ŌĆö With `on_behalf_of`, Stripe can send payment receipts from the connected account's brand. The app also sends booking confirmations via Resend. If both fire, the customer gets two emails. Fix: either disable Stripe receipt emails at the platform level (set `receipt_email: null` on PaymentIntent creation) or disable them in Express Dashboard settings. Verify which is the default for Express accounts ŌĆö Stripe may send receipts automatically unless explicitly suppressed.
- **Auto-poll timeout has no fallback state** ŌĆö Settings page State 2b (verifying) polls 12 times at 5s intervals (60s total). If Stripe verification takes longer (can be hours for some merchants), polling stops silently ŌĆö no error, no guidance. The pulsing animation implies active waiting but nothing is happening. Fix: after 12th poll, transition to a "still verifying" state with static copy: "Stripe is still reviewing your details. This can take up to a few hours. We'll update your dashboard when it's ready ŌĆö you can close this page." Remove pulsing animation, show static info block. Webhook handles the actual transition in the background.
- **Webhook event type misconfiguration can silently drop events** ŌĆö Both `/api/stripe/webhook` and `/api/stripe/connect-webhook` share the `processedStripeEvents` dedup table. If Stripe Dashboard config accidentally registers `account.updated` on both endpoints, whichever processes first wins ŌĆö and if the platform webhook gets it, it skips the event (no handler), marks it processed, and the Connect webhook never sees it. Account status update silently lost. Fix: both webhooks should log a warning when they receive an event type they don't handle (e.g., "Unexpected event type 'account.updated' at platform webhook ŌĆö check Stripe webhook configuration"). Surfaces misconfiguration immediately.
- **No minimum deposit floor** ŌĆö Tier pricing overrides can reduce `depositAmountCents` to any value, including below 50p (where the platform fee is waived). No platform-level minimum exists. Decision: enforce a ┬Ż1 (100p) platform minimum in `derivePaymentRequirement()` ŌĆö if `amountCents > 0 && amountCents < 100`, clamp to 100. This protects fee economics (50p on ┬Ż1 = 50% max effective rate) and prevents degenerate micro-deposits. The `topDepositWaived: true` path (sets to 0) should bypass the floor ŌĆö a waiver is an explicit merchant choice. Also add a `minimumDepositCents` to `shopPolicies` so merchants can set their own floor above the platform minimum.
- **Slot recovery bypasses Connect guard** ŌĆö `src/lib/slot-recovery.ts:530` hardcodes `paymentsEnabled: true` in the `acceptOffer()` call to `createAppointment()`. This means recovery bookings always attempt deposits regardless of Connect status. If Connect is not complete, the deposit goes to the platform account (not the merchant) ŌĆö money lands in the wrong place. Fix: derive `paymentsEnabled` from `shop.stripeOnboardingStatus === "complete"`, same as the booking page guard (spec 14). The shop is already available in `openOffer.shop`.
- **Disputes attributed to merchant with no visibility or notification** ŌĆö `on_behalf_of` makes the connected account the business of record, so disputes are the merchant's responsibility. But: (1) the merchant isn't told this during onboarding, (2) the platform isn't subscribed to `charge.dispute.created` events so has no visibility, (3) the merchant has no in-app dispute tools ŌĆö they'd need to find the Stripe Express Dashboard. For now parked. When addressed: subscribe to `charge.dispute.created` on Connect webhook, email/SMS the merchant with a dashboard link, and add expectation-setting copy to the Connect success state.
- **Kicksnare migration: pre-Connect appointments have no `depositSkipped` signal** ŌĆö All appointments created before the Connect migration have `depositSkipped = null`. The dashboard card Tier 2 count query (`depositSkipped = "connect_not_complete"`) returns 0 for these, so Kicksnare sees Tier 1 ("deposits aren't enabled yet") instead of Tier 2 with their actual booking count. Don't backfill ŌĆö it's revisionist. Accept that Tier 1 shows until Connect is complete, at which point the card disappears anyway. If this matters: add a fallback count of `depositSkipped IS NULL AND paymentStatus = 'unpaid'` for shops where `stripeAccountCreatedAt` is recent (first-time Connect adopters).
- **Payouts-not-enabled not surfaced in Connected state** ŌĆö A merchant can be `complete` (charges_enabled=true, details_submitted=true) but have `payouts_enabled=false` (bank details not yet verified by Stripe). Deposits collect into their connected account balance but nothing reaches their bank. Settings page State 3 should show a warning line when `chargesEnabled && !payoutsEnabled`: "Deposits are being collected. Stripe is still verifying your bank details for payouts ŌĆö check your Stripe Dashboard." Don't block deposits ŌĆö Stripe resolves this; just set expectations.
- **Re-engagement email window anchored incorrectly** ŌĆö The 24ŌĆō48h window is anchored to `stripeAccountCreatedAt` only. If the owner creates their Stripe account before setting up services, and adds services after 48h, the email never fires. Fix: anchor the window to the later of `stripeAccountCreatedAt` or "first service created date" (earliest `eventTypes.createdAt` for the shop). This decouples the two gates ŌĆö the time window adapts to late setup while the `hasServices` gate still prevents premature sends.
- **Stripe processing fee not recoverable on refund** ŌĆö Platform absorbs ~34p (Stripe 1.4% + 20p) per refunded Connect payment. This is acceptable at low volume but should be monitored. If refund rates are high, consider a cancellation fee policy instead. Not surfaced to merchants (deliberate ŌĆö it's a platform cost of business).
- **Slot button hover state not implemented** ŌĆö spec 05 calls for `#eeeeec` bg + lighter border on hover. Skipped to stay consistent with "design change only" and the V1-V3 pattern (no hover states added via JS). Add later if desired.
- **`bg-al-background` Tailwind class does not work** ŌĆö no `--color-al-background` mapping exists in `@theme inline`. Spec #12 references this class. Use `bg-al-surface` or `bg-background` instead (both resolve to `#f9f9f7`).
- **BookingNav shows static "Sign in" / "Start free trial"** regardless of session state. The current `SiteHeader` is session-aware (shows "Open App" / "Sign Out" when logged in). Decide if BookingNav needs session awareness.
- **Hardcoded hex values in 5 non-dashboard components** ŌĆö `attention-required-table.tsx`, `payments-table.tsx`, `curator-chip.tsx`, `customers-editorial.tsx`, `appointments-table.tsx` contain hardcoded hex values (`#ffdbcf`, `#2a170f`, `#fdd8cb`, `#e2bfb3`, `#380d01`, `#572411`) that should be replaced with DS tokens. Out of scope for Spec #11 (dashboard only). See `docs/shaping/home-screen-update/shapes/spike-hex-scope-beyond-dashboard.md`.
- **Deep Ledger `@theme inline` block still in globals.css** ŌĆö lines 3-160 define the full legacy `--color-*` token set. Annotated as "DEFERRED" pending landing page migration. Needs a dedicated cleanup wave to audit all remaining `--color-*` consumers and remove the block.
- **Hardcoded rgba hairlines in booking-form.tsx** ŌĆö 7 instances of `rgba(195,198,209,0.50)` should use `--al-hairline-strong`. Not a `--color-*` violation but a tokenization gap.
- **Hardcoded rgba warning border in conflict-alert-banner.tsx** ŌĆö `rgba(201,122,42,0.25)` at line 76 should use `--al-status-caution-border` (now defined in globals.css as of Wave 2 Slice 1). Not a `--color-*` violation but a tokenization gap.
- **`--font-display` and `--font-body` legacy aliases still in `@theme inline`** ŌĆö Wave 1 Slice 1 repointed them to Manrope; Wave 2 Slice 1 removed their usage from `@layer base` and utility classes. They now exist only as `@theme inline` definitions. 13 inline `var(--font-mono)` references across 4 component files (`booking-form`, `manage-booking-view`, `tier-policy-form`, `payment-policy-form`) still read from `@theme`. Consider removing the legacy aliases and migrating inline consumers to `--al-font-mono` in a future cleanup.
