- Pattern name: Extract-for-testability (pure function extraction)
- Problem it solves: Testing inline code embedded in large functions (e.g. inside `createAppointment()`) requires mocking the entire function's dependencies (DB, Stripe API, transaction context). The test setup is heavier than the logic being tested.
- Solution: Extract the inline logic into a pure exported function with explicit parameters. Test the pure function directly — zero mocks needed. Call the function from the original location via spread/destructure. The extraction is a net win: implementation code gets shorter (spread call replaces inline block), test code gets dramatically simpler, and the function becomes reusable if other callers need the same logic.
- First used in: webhook-unaware feature (Spec 04, `buildConnectPaymentMetadata()` in appointments.ts). Agent independently chose this approach during implementation — produced 11 focused tests vs the spec's expected 4, with zero mocks.
- Reusable? YES — any inline logic block in a large function that needs testing. Look for: metadata construction, status derivation, conditional object building. The pattern is especially valuable when the surrounding function has heavy I/O dependencies (DB, Stripe, email) but the logic being tested is pure data transformation.
