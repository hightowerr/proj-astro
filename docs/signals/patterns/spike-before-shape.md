- Pattern name: Spike before shape (confirmation spikes)
- Problem it solves: Specs make assumptions about field existence, query patterns, or infrastructure that may not hold. Discovered late, these cause rework.
- Solution: Run 2-3 targeted spikes against the actual codebase BEFORE finalizing the shape. Focus on: (1) does the assumed field/API exist? (2) what's the existing pattern for this infrastructure? (3) what's the exact code path being modified? (4) will the assumed code compile — check lint rules and TS types for the target file? (5) what types do target columns/fields accept? These are fast (minutes) and prevent incorrect spec assumptions from propagating through implementation. Items 4-5 added after webhook-unaware retro: all 7 drift deviations traced to codebase constraints (lint `no-console`, Stripe TS gaps, `Record<string, string>` column type) that existing spikes didn't check.
- First used in: stripe-connect (3 spikes: availabilityConfigured field, email template pattern, paymentsEnabled threading)
- Reusable? YES — any feature touching existing code should spike the assumed integration points
